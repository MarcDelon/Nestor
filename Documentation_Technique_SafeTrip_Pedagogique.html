1) Aperçu simple

SafeTrip est un site web pour réserver des voyages, discuter avec une agence et scanner des QR codes de billets/colis.

  • Ce que voit l’utilisateur: des pages rapides et jolies (Next.js/React) avec plusieurs langues.
  • Ce qui gère les données: un serveur (Express) qui parle à une base (PostgreSQL via Supabase).
  • Où c’est hébergé: Vercel pour le site, Supabase pour la base, un service pour l’API.

2) Frontend (côté utilisateur)

Next.js (framework React)

Définition simple: Next.js est un outil pour construire des sites rapides avec React.

Pourquoi ce choix ? Parce que…

  • Le site se charge vite parce que certaines pages sont préparées à l’avance sur le serveur (meilleur SEO et temps d’affichage).
  • Le déploiement est très facile sur Vercel parce que Next.js et Vercel sont faits pour bien travailler ensemble.
  • La navigation est fluide parce que Next découpe le code automatiquement (on ne télécharge que ce qu’il faut).

Alternatives: Create React App (plus ancien, moins optimisé), Gatsby (très orienté génération statique), Remix (plus technique côté serveur).

Exemple concret: la page d’accueil s’affiche rapidement et Google la comprend bien (meilleur référencement).

TypeScript

Définition simple: c’est du JavaScript avec des « types » pour éviter des erreurs.

Pourquoi ? Parce que…

  • On repère des fautes plus tôt parce que l’éditeur signale les incompatibilités (ex: additionner un texte et un nombre).
  • Le code est plus clair pour l’équipe parce que chaque variable a une forme attendue.

Alternative: JavaScript pur (plus simple au début, mais erreurs plus tardives et plus coûteuses).

Exemple: quand on manipule un « billet », TypeScript garantit qu’il contient bien les champs attendus (date, statut…).

CSS Modules

Définition simple: une manière de styliser chaque composant sans mélanger les styles entre eux.

Pourquoi ? Parce que…

  • Les styles restent propres parce que chaque fichier CSS ne touche qu’un composant.
  • Pas de dépendance lourde au runtime (performant).

Alternatives: Tailwind (rapide mais plus de classes dans le HTML), Styled‑Components (agréable mais ajoute un runtime).

Exemple: la couleur d’un bouton dans le chat n’affecte pas les autres boutons ailleurs.

next-intl (traductions)

Définition simple: outil pour afficher le site en plusieurs langues.

Pourquoi ? Parce que…

  • Il s’intègre bien à Next.js (pages préparées côté serveur).
  • On peut ajouter des langues facilement à l’avenir.

Alternative: i18next/react-intl (très bien aussi, intégration un peu plus manuelle).

jsQR + getUserMedia + Canvas (scan QR)

Définition simple: on ouvre la caméra (getUserMedia), on lit l’image (Canvas) et jsQR trouve le QR code.

Pourquoi ? Parce que…

  • Tout se passe dans le navigateur parce que c’est plus rapide (pas d’envoi vidéo au serveur).
  • C’est léger et fiable pour nos QR (billets/colis).

Alternative: ZXing (similaire, parfois plus lourd), service externe (latence et dépendance réseau).

Important mobile: les navigateurs bloquent la caméra si le site n’est pas en HTTPS. Donc en production, on passe par Vercel (HTTPS), parce que c’est une exigence de sécurité du navigateur.

3) Backend (serveur)

Express (TypeScript)

Définition simple: un petit serveur qui reçoit des requêtes (URL) et renvoie des réponses (JSON).

Pourquoi ? Parce que…

  • C’est simple et connu, donc on va vite.
  • On garde le contrôle précis sur chaque route (messages, profils, scan…).

Alternatives: NestJS (très structuré, plus lourd au début), Fastify (plus rapide mais API différente).

Exemple: quand l’appli demande « donne-moi les messages du fil X », Express lit la base et renvoie la liste.

4) Base de données (Supabase / PostgreSQL)

Définition simple: Supabase héberge une base PostgreSQL (tables, lignes, colonnes) pour nous.

Pourquoi PostgreSQL via Supabase ? Parce que…

  • C’est du SQL classique (puissant et standard), donc facile à requêter.
  • Supabase gère l’hébergement, les sauvegardes et la sécurité de base.

Alternatives: Firebase (NoSQL, très bien mais modèle différent), PlanetScale (MySQL), MongoDB (NoSQL).

Exemple: la table messages contient: texte, heure, auteur, et aussi reply_to_id/reply_to_text pour afficher « Réponse à … ».

5) Messagerie (chat)

Fonctions: envoyer, modifier, supprimer un message; marquer comme lu; répondre à un message avec aperçu.

Comment ça marche simplement:

  • Le site demande la liste des messages toutes les 3 secondes (polling) parce que c’est simple et fiable.
  • Quand on répond, on enregistre l’ID et un extrait du message d’origine pour l’afficher dans la bulle.

Pourquoi pas WebSockets pour l’instant ? Parce que… pour notre charge actuelle, le polling est suffisant et plus rapide à mettre en place. On pourra passer à du temps réel poussé quand nécessaire.

6) Scan QR (caméra)

  • On ouvre la caméra (avec l’autorisation de l’utilisateur) et on analyse l’image en direct.
  • HTTPS obligatoire sur téléphone parce que les navigateurs protègent l’accès caméra. Vercel fournit HTTPS automatiquement.

7) Hébergement (Vercel) & déploiement

Vercel (Frontend)

  • Déploiement automatique depuis GitHub parce que Vercel se connecte directement au dépôt.
  • Le site est servi depuis des serveurs proches des utilisateurs (CDN) parce que cela réduit le temps de chargement.
  • Les prévisualisations de chaque Pull Request aident à valider avant mise en prod.

Backend: on choisit un service (Render, Railway, Fly.io) qui lance notre serveur Express. Parce que c’est simple et économique.

8) Collaboration (GitHub)

  • On pousse le code sur des branches (ex: feature/chat-reply).
  • On ouvre une Pull Request pour discuter et vérifier.
  • Vercel crée automatiquement un lien de test pour cette PR (préview).

Pourquoi GitHub ? Parce que… c’est l’outil standard du marché, simple à partager et très bien intégré à Vercel.

9) Sécurité & performances

  • Sécurité: HTTPS, vérification d’authentification (middleware), validation des données en entrée.
  • Performances: Next.js prépare les pages côté serveur; images optimisées; code découpé.
  • Observabilité: logs serveur; on peut ajouter Sentry pour repérer les erreurs en production.

10) Tests & qualité

  • Tests simples pour s’assurer que les routes essentielles répondent bien.
  • Tests de scénario (E2E) envisagés pour la messagerie et le scan.
  • Formatage/Lint pour garder un code propre et homogène.

11) Mini‑glossaire (mots clés)

Framework
Boîte à outils qui accélère le développement (ex: Next.js pour React).
API
Porte d’entrée pour discuter avec le serveur (ex: /api/agency/messages).
Base de données
Endroit où l’on enregistre les informations (ex: messages, billets, colis).
HTTPS
Connexion sécurisée (obligatoire pour la caméra sur mobile).
CDN
Réseau de serveurs proches des utilisateurs pour charger plus vite.
