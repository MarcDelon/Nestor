1) Aperçu du projet

SafeTrip est une application web pour voyageurs et agences: réservation, suivi colis/bagages, messagerie agence–client, et scan QR pour billets/étiquettes. L’interface est responsive et multilingue.

  • Frontend: Next.js (React, TypeScript), CSS Modules, next-intl.
  • Backend: Express (TypeScript), endpoints REST.
  • Base: Supabase (PostgreSQL managé).
  • Déploiement: Vercel (frontend). Collaboration: GitHub.

2) Architecture & flux

Diagramme (vue logique)

[ Client (Next.js) ]  ←→  [ API Express ]  ←→  [ Supabase (PostgreSQL) ]
       ↑   |   fetch/polling           | SQL / RPC / Auth
       |   ↓                          
   Caméra (getUserMedia)               
   jsQR (décodage)

Flux typiques

  • Messagerie: le client envoie un message → API Express → enregistrement Postgres → polling côté client (3s) + marquage lu.
  • Scan QR: la caméra (getUserMedia) capture une frame → jsQR décode → affichage + actions (ex: notification de livraison).

3) Outils & Technologies

Frontend

  • Next.js – Framework React (SSR/SSG, App Router). Pourquoi: perfs, SEO, intégration native Vercel. Alternatives: Gatsby, Remix, CRA (déconseillé).
  • TypeScript – Typage statique. Pourquoi: sécurité, refactoring. Alt.: JS pur (moins sûr), Flow (peu adopté).
  • CSS Modules – Styles scopés. Pourquoi: simple, zéro runtime. Alt.: Tailwind (utilitaires), CSS-in-JS.
  • next-intl – i18n côté Next. Pourquoi: App Router friendly. Alt.: i18next/react-intl.
  • jsQR + getUserMedia + Canvas – Décodage QR en local. Pourquoi: léger, sans serveur vision. Alt.: ZXing, backends ML.

Backend

  • Express (TypeScript) – Micro‑framework HTTP. Pourquoi: léger, connu. Alt.: NestJS (opinionné), Fastify (performant).
  • Supabase (PostgreSQL) – BaaS SQL. Pourquoi: SQL robuste, hébergé, intégrations. Alt.: Firebase (NoSQL), PlanetScale (MySQL), MongoDB.

Collaboration & Hébergement

  • GitHub – Hébergement code, PR, Issues, Projects. Pourquoi: standard, intégration Vercel.
  • Vercel – Hébergement frontend Next. Pourquoi: DX, CDN, prévisualisations PR. Alt.: Netlify.

4) Frontend (Next.js)

  • Routage App Router; pages clients/agences (dashboards), composant de messagerie, scanner QR.
  • Styles en CSS Modules pour encapsulation et cohérence.
  • i18n via next-intl.
  • Messagerie: affichage threads, polling 3s (configurable), marquage lu.
  • Réponse à un message: aperçu « Réponse à … » avec reply_to_id/reply_to_text.

6) Base de données (Supabase/PostgreSQL)

Table messages (extraits):

  • id (PK), agency_id, thread_id, sender ('agency'|'contact')
  • text, time, is_read (bool)
  • reply_to_id (int, optionnel), reply_to_text (text, optionnel)
  • created_at (timestamp)

Migrations fournies (schema.sql + migrations/*). Certaines tables désactivent RLS selon besoins.

7) Messagerie & Statuts

  • Polling (≈3s) pour récupérer les messages d’un thread.
  • Marquage « lu » via PUT /messages/:threadId/read.
  • Édition/Suppression possibles (côté auteur; confirmation contextuelle).
  • Réponses: les champs reply_to_id/reply_to_text pilotent l’aperçu « Réponse à … ».

8) Scan QR & Traçabilité

  • Caméra: navigator.mediaDevices.getUserMedia (mobile → nécessite HTTPS ou localhost).
  • Décodage: jsQR à partir d’une frame Canvas.
  • UI: aperçu, animation ligne de scan, actions (ex: notifier livraison).

Important mobile: sur IP locale en HTTP, l’accès caméra est souvent bloqué. Utiliser HTTPS (Vercel) ou un tunnel (ngrok/Cloudflare) pour tester sur téléphone.

9) Hébergement (Vercel) & Backend

Frontend – Vercel

  1. Connecter le repo GitHub → Importer le projet (dossier frontend s’il est séparé).
  2. Définir les variables d’env.: NEXT_PUBLIC_API_URL (pointe vers l’API), i18n si besoin.
  3. Build par défaut Next.js → déploiement auto (Preview sur PR, Production sur merge).

Backend – options

  • Service Express: Render, Railway, Fly.io (simples et économiques).
  • Alternative: Serverless Functions (Vercel/Cloudflare) après adaptation des routes.
  • Base: Supabase gère PostgreSQL + Auth + Stockage (déjà en place).

11) Sécurité, Performance, Observabilité

  • HTTPS, CORS, validation d’entrée, requireAuth sur endpoints sensibles.
  • Perfs: SSR Next, code splitting, images optimisées, caches.
  • Observabilité: logs serveur; recommandation Sentry/Logtail pour prod.

12) Tests & Qualité

  • Unitaires (utils, mappages), intégration (routes clés).
  • E2E (Playwright/Cypress) pour parcours critiques (messagerie, scan).
  • Lint/Format (ESLint/Prettier) – recommandés.

13) Checklist de déploiement

  • Supabase: exécuter les migrations DB (schema.sql + migrations/..).

  • Frontend: Vercel relié au repo; configurer NEXT_PUBLIC_API_URL.
